# Examplan — планировщик подготовки к экзамену

Веб-приложение на Go, которое строит оптимальный план подготовки к экзамену,
используя простую математическую модель из математического анализа:

- **Экстремум** — поиск оптимального числа часов занятий в день (вершина
  колоколообразной функции эффективности `E(x)`).
- **Производная** — скорость усвоения знаний `K'(t)`, убывающая со временем
  (закон убывающей отдачи).
- **Интеграл** — накопленные знания `K(t)` как сумма всех дневных приростов
  `K'(t)` с начала подготовки.

## Стек

- **Backend:** Go (`net/http`, без фреймворков), вся математика считается
  на сервере (`internal/calc`).
- **Хранилище:** SQLite (`internal/db`) — история сохранённых расчётов.
- **Frontend:** чистый HTML/CSS/JS, графики рисуются на `<canvas>` без
  внешних библиотек (`static/js/charts.js`).

## Запуск

Понадобится Go 1.22+ и компилятор C (для `go-sqlite3`, который использует cgo).

```bash
cd examplan
go mod tidy        # подтянет зависимость github.com/mattn/go-sqlite3
go run main.go
```

Сервер поднимется на `http://localhost:8080`.

Если у вас Windows и нет gcc — самый простой вариант: поставить
[MSYS2/MinGW](https://www.mingw-w64.org/) и включить `CGO_ENABLED=1`,
либо заменить `mattn/go-sqlite3` на чистый Go-драйвер
`modernc.org/sqlite` (drop-in замена с минимальными правками `db.go`).

### Переменные окружения (опционально)

| Переменная       | По умолчанию     | Назначение                    |
|------------------|------------------|--------------------------------|
| `PORT`           | —                | если задана, имеет приоритет — её использует Render и другие PaaS |
| `EXAMPLAN_ADDR`  | `:8080`          | адрес, на котором слушает сервер (если `PORT` не задана) |
| `EXAMPLAN_DB`    | `examplan.db`    | путь к файлу базы данных SQLite |

## API

### `POST /api/calculate`

Запрос:
```json
{
  "subject": "Математика",
  "daysLeft": 30,
  "freeHours": 6,
  "difficulty": 3,
  "currentLevel": 40,
  "save": true
}
```

Ответ: объект `result` с полями `optimalHours`, `predictedResult`,
`burnoutRisk`, `burnoutMessage`, `efficiencyCurve`, `knowledgeCurve`,
`totalKnowledge`, плюс `planId`, если `save: true`.

### `GET /api/history`

Возвращает последние 30 сохранённых расчётов.

### `DELETE /api/history/delete?id=N`

Удаляет запись истории по ID.

## Деплой на Render (бесплатно, без карты)

Render — единственная популярная PaaS-платформа, у которой по состоянию на
2026 год остался реально бесплатный веб-сервис без привязки карты. Минус:
сервис «засыпает» после 15 минут без запросов и просыпается ~30-60 секунд
при следующем заходе. Для учебного проекта это нормально.

> ⚠️ На бесплатном тарифе Render файловая система эфемерна (persistent disk
> доступен только на платных планах). Это значит, что файл `examplan.db`
> будет обнуляться при каждом перезапуске/«просыпании» сервиса — история
> расчётов не сохранится надолго. Сами расчёты при этом продолжат работать
> корректно, это не баг.

### Шаг 1 — выложить код на GitHub

1. Зарегистрируйся на [github.com](https://github.com), если ещё нет аккаунта.
2. Создай новый репозиторий (например `examplan`), без README/.gitignore —
   они уже есть в проекте.
3. Залей в него содержимое папки `examplan/` (через GitHub Desktop, веб-загрузку
   файлов на странице репозитория, или `git push`, если знаком с консолью).

### Шаг 2 — создать сервис на Render

1. Зарегистрируйся на [render.com](https://render.com) (можно через GitHub —
   так сразу появится доступ к репозиториям).
2. На дашборде нажми **New +** → **Web Service**.
3. Выбери **Build and deploy from a Git repository**, подключи свой GitHub-аккаунт
   и выбери репозиторий `examplan`.
4. В форме настройки укажи:
   - **Name** — любое (например `examplan`)
   - **Region** — ближайший к тебе
   - **Branch** — `main`
   - **Runtime** — Render сам определит **Docker** по наличию `Dockerfile`
     в корне репозитория
   - **Instance Type** — **Free**
5. Жми **Create Web Service**.

Render соберёт Docker-образ по `Dockerfile` (это займёт пару минут — собирается
с нуля, включая компиляцию cgo-зависимости sqlite3) и выдаст публичный адрес вида
`https://examplan.onrender.com`.

### Шаг 3 — проверить

Открой выданный адрес — должна загрузиться главная страница планировщика.
Если сервис «уснул», первая загрузка может занять полминуты — это нормально,
дальше будет работать быстро, пока не пройдёт 15 минут без запросов.

### Если деплой упал с ошибкой

Зайди на вкладку **Logs** сервиса в дашборде Render — там видно, на каком этапе
(build/deploy/runtime) что-то пошло не так. Самые частые причины:
- забыли закоммитить `Dockerfile`, `go.mod` или `go.sum` в репозиторий;
- неверно указана ветка (Render по умолчанию следит за `main`).



Тема — «научная тетрадь + дашборд»: графитовый фон с миллиметровкой,
бумажные карточки, янтарный акцент для кривой эффективности (экстремум) и
индиго для кривой роста знаний (интеграл). Шрифты: Space Grotesk
(заголовки), Inter (текст), JetBrains Mono (цифры и формулы).

> в CSS-переменные в начале файла).
